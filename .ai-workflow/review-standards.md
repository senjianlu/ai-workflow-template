# 评审标准(评审者手册)

本文件是本项目代码评审的唯一权威标准,由评审指令
(.ai-workflow/prompts/review.md)显式引用。

评审者是本项目的独立代码评审者:只评审,不修改任何代码,也不在工作区
新建或改动任何文件(只读沙箱机器强制,任何写入都会失败);可执行的
动作以下方「评审动作边界」为准。

## 评审输入

- 当前工作区的未提交改动(含新增文件)
- 本轮任务的 plan.md(路径由评审指令给出)

核心问题永远是两个:实现是否忠于 plan?plan 中的测试用例是否真实覆盖并通过?

## 评审动作边界

评审在**只读沙箱**中直审原仓库(机制见 docs/decisions/0005),任何写入
都会被沙箱拒绝;评审动作以此为前提:

- **允许**:读取代码与 diff、grep 类检索;运行**确无写入副作用**的
  只读检查(如 `tsc --noEmit`、禁用缓存的 lint)。命令因只读沙箱的
  写失败而报错时,**不视为代码缺陷**,该项转证据核验并在评审输出中
  注明"未能在边界内核验"。
- **不运行任何测试**(含单元测试——多数 runner 需写缓存,只读下天然
  失败;测试核验整体走下方证据协议)。
- **超时与中止**:单条命令设超时上限(默认 5 分钟,建议用 timeout 等
  机制强制),超时即中止并在评审输出中记录"未能在边界内核验",不得
  反复重试拉长评审;疑似不自终止的命令一律不执行。
- **禁止**:安装或更新依赖(npm / pnpm install、pip 等)、构建
  (next build 等)、启动任何服务器或长驻进程(next dev、uvicorn、
  docker 等)、E2E(playwright)、访问网络。只读下多数天然失败,
  保留为行为约束:不得尝试。
- **证据协议**:测试真实性核验一律走证据,**且逐条按 plan.md 声明的证据
  档位执行**,不得对全部用例统一套用最严标准。档位定义、约束与逐档核验
  规则见下方「测试证据档位」小节。

## 测试证据档位(证据契约)

证据形态在 **plan 阶段**即以契约形式定好:plan.md 的测试用例表逐条声明
「档位」与「证据形态」,实现者按契约留证,评审者按契约核验。三档定义:

| 档 | 适用范围 | 证据形态 |
|---|---|---|
| **A** | 可由非交互命令判定的用例(单元测试、E2E runner、lint、tsc、grep/静态校验、脚本) | A 档须留完整原始输出:执行命令 + 完整 stdout/stderr + 退出码,落任务目录 `evidence/` |
| **B** | 靠静态事实即可确证的断言(配置生效、文案键存在、路由已注册) | 日志片段,或代码/文档引用——引用**须带 `文件:行号`**,供评审自行 grep 复核 |
| **C** | 确实无法自动化的人工交互验证(真机视觉、剪贴板、系统弹窗、跨设备) | 实现者陈述 + 关键锚点(相关代码位置、截图文件名、控制台片段) |

档位约束:

- **A 档强制**:凡可自动化者一律 A 档,**不得降档**。plan 中把可自动化项
  写成 C 档 → major。
- **C 档上限**:C 档条目数 ≤ 该 plan 测试用例总数的 1/3(向下取整);
  **总数 < 3 时至多 1 条**。超限 → major。每条 C 档须在 plan 中写明
  **为何不可自动化**,缺理由 → major。
- **形态优先级**:日志/文本证据优先于截图。截图只能作为 C 档的补充锚点,
  **不得作为任一用例的唯一证据**——评审在只读沙箱中无法读取图像。
- **向后兼容**:plan **未声明档位**的用例(本约定之前的历史任务)一律
  **按 A 档处理**。

### 逐档核验规则

评审前先读 plan.md 测试用例表的「档位」列,**逐条按该列核验**,不得对全部
用例统一套用 A 档要求:

- **A 档**:`evidence/` 中须有该用例的完整原始输出;**缺失或不完整 →
  blocker**,且 detail 中**必须列明需要补交的证据清单**(具体到用例编号与
  所缺内容:命令、输出、退出码等),供实现者按单补交后进入下一轮。
- **B 档**:核验引用的 `文件:行号` 或日志片段——**引用不存在 → blocker**;
  **引用存在但不支持结论 → major**。
- **C 档**:**默认采信**实现记录中的结果描述。仅在下列三种情形判问题:
  1. 描述与 plan 的预期结果或其他证据**自相矛盾**;
  2. 该用例在 plan 中**并非 C 档**(即实现阶段被擅自降档);
  3. **关键锚点缺失**,使描述完全无法定位。

  **不得仅以「无原始命令输出」为由对 C 档判 blocker。**
- **任一档位**:证据与实现记录**矛盾** → 按虚报测试处理(blocker)。

## 严重度定义(唯一权威来源)

- **plan-blocker**:方案/架构层面的错误,修改代码无法解决,必须退回方案阶段
- **blocker**:实现层错误,导致功能不可用、数据损坏或安全漏洞
- **major**:实现层缺陷,功能可用但存在正确性或健壮性问题,必须修复
- **minor**:风格、可读性、边缘优化,可修可不修

## 判定规则

以下规则适用于**实现层评审**(review.sh,对已实现的代码 diff):

- 存在任一 blocker 或 major → **fail**
- 仅有 minor 或无问题 → **pass**
- plan-blocker 单独列出,不计入 pass/fail,但必须置顶显著标注

## plan 阶段评审(plan-review)语义

**plan 阶段评审**(plan-review.sh,实现前审 plan.md)另用一套判定:此时尚无
实现,**不使用 blocker**;判定规则为——

- 存在任一 **plan-blocker 或 major → fail**(须就地修订 plan.md 后重评)
- 仅有 minor 或无问题 → **pass**

即:同一 plan-blocker 严重度,在实现层评审中不计入 pass/fail(单独升级),在
plan 阶段评审中直接计入 fail——因为 plan 阶段的全部目的就是拦截 plan-blocker。
两套规则分域适用,互不覆盖。
