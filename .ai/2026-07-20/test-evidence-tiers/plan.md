# 方案:测试证据契约改为「档位制」

## 背景与目标

现状:`review-standards.md` 的证据协议是「全部测试用例统一适用」,要求每条
用例都在 `evidence/` 留完整原始命令输出(命令 + stdout/stderr + 退出码),
缺失即 blocker。而 `templates/plan.md` 的测试用例表只有「预期结果」,不声明
「这条怎么验、留什么证据」。

后果:证据形态在 plan 阶段没有契约,评审端只能按最严标准(A 类原始输出)
统一想象。凡是靠视觉、剪贴板、系统弹窗等人工交互验证的用例,实现者只能交
截图或手工汇总,必然被判「证据缺失」而 blocker 打回——且截图在只读沙箱里
评审者根本无法读取,是最弱的一种证据。这不是评审跑偏,是规则本身缺一层。

目标:把证据契约前移到 plan,按**证据档位**分类核验,而不是一刀切。

- 凡可自动化的用例强制 A 档(留原始输出,成本近乎为零,不得降档)
- 靠静态事实即可确证的用例走 B 档(日志片段 / 代码引用带文件:行号)
- 只有确实无法自动化的走 C 档(实现者陈述 + 锚点),且有数量上限与理由要求
- 评审对 C 档默认采信,不得仅以「无原始命令输出」判 blocker
- 实现阶段禁止自行降档或改写用例重要性;测不了如实标 `blocked` 并停下问用户

明确不做:不引入「需证据项 ≤ 总数 50%」的比例配额。比例与真实性无因果关系,
且可通过拆细用例稀释分母;约束应落在「该自动化却偷懒」上(已与用户确认)。

## 实现方案

### 1. 证据档位定义(写入 review-standards.md,为唯一权威)

| 档 | 适用 | 证据形态 |
|---|---|---|
| **A** | 可由非交互命令判定的用例(单测、E2E runner、lint、tsc、grep/静态校验、脚本) | 完整命令 + 完整原始 stdout/stderr + 退出码,落 `evidence/` |
| **B** | 靠静态事实即可确证的断言(配置生效、文案键存在、路由已注册) | 日志片段,或代码/文档引用**须带 `文件:行号`**,评审可自行 grep 复核 |
| **C** | 确无法自动化的人工交互验证(真机视觉、剪贴板、系统弹窗、跨设备) | 实现者陈述 + 关键锚点(相关代码位置 / 截图文件名 / 控制台片段) |

约束:

- **A 档强制**:凡可自动化者一律 A,不得降档;plan 阶段把可自动化项写成
  C 档 → major。
- **C 档上限**:C 档条目数 ≤ 该 plan 测试用例总数的 1/3(向下取整);总数
  < 3 时至多 1 条。超限 → plan 阶段 major。每条 C 档须在 plan 中写明
  **为何不可自动化**,缺理由 → major。
- **证据形态优先级**:日志/文本 > 截图。截图只能作为 C 档的补充锚点,
  **不得作为任一用例的唯一证据**——评审在只读沙箱中无法读取图像。
- **向后兼容**:plan 未声明档位的用例(本改动之前的旧任务)按 A 档处理。

### 2. 证据核验规则(替换现行 review-standards.md 第 35–40 行)

评审前先读 plan.md 测试用例表的「档位」列,**逐条按该列执行核验,不得对
全部用例统一套用 A 档要求**:

- **A 档**:`evidence/` 须有完整原始输出。缺失或不完整 → blocker,detail
  必须列明补交清单(具体到用例编号与所缺内容)。
- **B 档**:核验引用的 `文件:行号` 或日志片段是否真实存在、是否支持结论
  (评审可自行 grep 复核)。引用不存在 → blocker;存在但不支持结论 → major。
- **C 档**:**默认采信实现记录中的结果描述**。仅在下列情形判问题:
  1. 描述与 plan 的预期结果或其他证据**自相矛盾**;
  2. 该用例在 plan 中**并非 C 档**(即实现阶段被擅自降档);
  3. **关键锚点缺失**,使描述完全无法定位。

  **不得仅以「无原始命令输出」为由对 C 档判 blocker。**
- 任一档位:证据与实现记录**矛盾** → 虚报测试,blocker。

### 3. 实现阶段硬规则(写入 rawf-implement SKILL)

- 按 plan 声明的档位留证据,不按自己的判断增减。
- **不得下调 plan 已定的档位,不得改写用例的重要性或预期结果**。实测发现
  某用例无法按原档位执行 → 结果记 `blocked`,**停下向用户说明**,由用户
  决定改 plan 还是接受降档;不得自行降档后照常送审。
- **禁止虚构或推测测试结果**。未执行写「未执行」+ 原因,失败写失败。
- 存在 `blocked` 项时不进入评审,须先经用户裁决。

### 4. 决策文档回写(docs/decisions)

现行 [`0005-review-readonly.md`](../../../docs/decisions/0005-review-readonly.md)
的「决定」写有「全部测试用例**统一**走证据打回协议」,「影响」写有
「rawf-implement 第 5 步升级为**所有用例**完整原始输出强制落盘」——本方案正是
在修订这两条,不同步回写会让 `docs/` 的持久真相与 `review-standards.md` 互斥
(AGENTS.md 硬规则要求收尾前回写,与代码同一提交)。按
`docs/decisions/README.md`「推翻要新增记录 + 旧文件顶部标注取代,不删历史」:

- **新增 `0006-test-evidence-tiers.md`**:四段式记录。背景写明一刀切协议在
  人工交互类用例上必然误判(本任务缘起);决定写明 A/B/C 三档、A 档强制、
  C 档 1/3 上限与采信条款、截图不得为唯一证据、未声明档位回退 A 档,并列出
  被否掉的备选(比例配额制:与真实性无因果、可拆细稀释分母);影响写明
  0005 的「统一/所有用例」条款被本条**部分取代**,只读沙箱、不重跑测试、
  指纹双保险等其余决定继续有效。
- **标注 0005**:**仅在顶部**加一行「证据协议部分已被 0006 取代(见
  docs/decisions/0006-test-evidence-tiers.md)」。**正文一字不改**——包括
  上述两处「统一/所有用例」表述:它们是 2026-07-18 当时的真实决定,属历史
  记录。按 `docs/decisions/README.md:17-18`「不删除历史」与 0004 的既有先例
  (顶部标注取代、正文原样保留),推翻旧决策靠新增记录 + 顶部标注,而非
  改写旧正文。互斥感由顶部标注消解:读者读到 0005 时即被导向 0006。

## 风险与回滚

- **自举效应**:`review.sh` 评审的是工作区未提交改动,而评审指令要求评审者
  「评审前先完整阅读 review-standards.md」——它读到的是**改动后**的版本,
  因此新规则在本任务自己的评审轮即生效。风险面:若新条款写得自相矛盾,本轮
  评审可能被自身规则绊住;缓解是三档定义与判定规则集中写在同一小节,并由
  TC-10 / TC-11 反向校验旧表述已彻底删净,避免双份权威并存。注意本任务全部
  用例均为 A 档,自举不影响验收——C 档语义由 TC-04 静态覆盖。
- **旧句残留(核心风险)**:本改动的失败模式不是「新条款没写」,而是「旧条款
  没删」——三处现存硬编码(review-standards 的「全部测试用例统一适用」、
  prompts/review.md 的无条件 blocker 句、rawf-implement 的「所有测试用例的
  完整原始输出…必须」)若只增不删,评审与实现侧会继续按 A 档处理全部用例,
  方案目标完全落空。TC-10(指名三处原句)与 TC-11(全域扫描未被 A 档限定的
  强制)是该风险的专项拦截。
- **放松过度**:C 档采信条款可能被后续任务滥用,把该自动化的项塞进 C 档。
  缓解:A 档强制 + C 档 1/3 上限 + 逐条理由,三重约束都落在 plan 阶段,由
  plan-review 拦截;实现阶段的「不得降档」堵住事后规避。
- **旧任务兼容**:`.ai/` 下已有 4 个历史任务目录,其 plan 无档位列。回退条款
  (未声明 → 按 A 档)保证语义不变。
- **决策沿革**:0005 是本改动的直接前置决策,只部分取代其证据协议条款,
  不动其只读沙箱、不重跑测试、指纹双保险等其余决定;误把 0005 整条作废会
  连带推翻只读评审机制。0006 中须显式限定取代范围,TC-14 校验沿革标注到位。
- **改动覆盖不均**:三个 SKILL.md 中 implement 侧改动最重、最易被记住,而
  plan 与 review 侧的正文改动易被漏掉——它们分别决定「档位在 plan 阶段是否
  被声明」与「评审误判是否被申诉」,漏改即整条链路回落旧语义。TC-12/TC-13
  逐个 skill 正文校验,补上 TC-07(仅验 frontmatter)的盲区。
- **回滚**:纯文档/配置改动,`git checkout -- .ai-workflow .claude/skills docs/decisions` 即可
  完全还原,无数据与运行时影响。
