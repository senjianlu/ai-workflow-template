# rawf-plan:方案阶段

## 步骤

1. 与用户对齐需求。有歧义先问,不要带着猜测写方案。
2. 创建任务目录 `.ai/<yyyy-mm-dd>/<task-slug>/`(日期用今天,slug 用简短
   kebab-case 英文);若同名目录已存在,询问用户是续做还是换名。同时在其下
   建两个固定子目录并各放 `.gitkeep` 占位(空目录才能随任务入库):
   `evidence/`(证据:测试输出、截图、验证记录)与 `assets/`(资源:Design
   稿、参考图、外部素材)。
3. 将任务目录相对路径(如 `.ai/2026-07-05/add-login`)写入 `.ai/.current-task`。
4. 调研代码库后,复制 `.ai-workflow/templates/plan.md` 到任务目录并填写,
   保持 `status: draft`。
5. 测试用例章节为硬要求:每条含编号 TC-NN、前置条件、步骤、预期结果,
   且至少一条覆盖异常/边界路径。没有可执行的验收标准的方案不算方案。
   **证据契约同为硬要求**:每条用例须**逐条声明档位与证据形态**(档位
   权威定义见 .ai-workflow/review-standards.md「测试证据档位」)——
   - **可自动化者一律 A 档、不得降档**:凡能由非交互命令判定(测试、
     lint、tsc、grep、脚本)的用例都必须走 A 档;把这类用例写成 C 档
     会在 plan 评审中判 major;
   - **C 档 ≤ 用例总数 1/3 且须写明不可自动化的理由**(总数 < 3 时至多
     1 条);上限是上限不是下限,没有人工交互项时 C 档为 0 才是常态;
   - 证据形态优先取日志/文本;截图不得作为任一用例的唯一证据。
6. 【plan 评审闸,仅当本方案预计触及文件 > 10 时执行(≤ 10 不走此闸),
   在确认闸之前】
   运行 `bash .ai-workflow/scripts/plan-review.sh`(**后台方式**并等待完成,
   评审耗时波动大)。退出码 3 = 执行异常,把 stderr 报用户后停下。按结果分流:
   - fail(有 plan-blocker 或 major)→ **就地修订 plan.md**(尽量一次改完),
     重跑评审;脚本强制轮次上限(**默认 3 轮**),达上限(exit 3 提示
     "N 轮上限")仍不收敛 → 停止,把分歧交用户敲定,不自行绕过。
   - pass(仅 minor / 无问题)→ 继续。
   拿不准是否采纳某条意见时,不"解释掉"后静默忽略,原样呈用户定夺。
   轮次上限仅当**用户明确要求放宽**时,才在 plan.md frontmatter 写
   `plan_review_max_rounds: <正整数>` 覆盖默认值,并在第 7 步摘要中声明;
   不得未经用户提出而自行写入或建议性写入该字段。
7. 向用户呈现摘要:目标、改动范围、测试用例清单、主要风险;若跑过 plan 评审,
   附评审结论与所用轮次。明确请求确认。
8. 【确认闸】仅当用户明确表示同意后:将 plan.md 的 status 改为 approved,
   填写 approved_at(日期时间)。用户要求修改则改完回到第 7 步。
9. 在 status 变为 approved 之前,不进行任何源码改动(有 hook 强制)。
