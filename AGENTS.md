# AGENTS.md

## 用户交互规范
- 所有面向用户的说明、提问、进度更新和最终答复均使用简体中文。
- 代码、命令、路径、标识符、配置字段和必须保持准确的原文可以保留其原始语言；相关解释使用中文。
- 表达应简洁、直接，先给结论，再补充必要的原因或步骤。
- 不重复用户已经提供的信息，不展开与当前任务无关的背景。
- 存在多个合理方案时，提供 2～3 个选项：
  - 将推荐方案放在第一位，并明确标注“推荐”。
  - 简要说明每个方案的主要收益、代价和适用条件。
  - 不提供没有实际区别的虚假选项。
- 仅在存在实质性权衡（涉及产品方向、明显风险、不可逆操作或成本差异）时提供选项；无实质性权衡时采用合理默认值。
- 提问时一次只集中询问当前阻塞工作的关键问题；能够继续安全推进的部分应继续推进。

## 角色与协作
本项目由 Claude Code 负责方案与实现，Codex 负责独立评审；本节规定双方分工与交接协议。

- **实施者 = Claude Code**：负责方案制定与编码实现。角色/模型定义见 `.claude/agents/`。
- **评审者 = Codex**：只读，负责对改动做跨模型评审，不修改代码，仅可以修改指定的评审文档。具体评审规则见本文件“严重度定义”和“工作流”。

### 步骤 0 · 准入判定（动手前判定一次）
满足以下**任一**条件即进入完整工作流：
- 用户显式要求走 AI 工作流(消息含 `参照 AI 工作流` 或等价明确指令)
- 本次改动预计触及（新增/修改/删除）文件数 **> 5**

两者都不满足时，由当前会话模型直接修改、不走流程。
> 若走直改路径、但实现中实际触及文件数超过 5，立即停止，转入下方工作流并补出方案。

### 严重度定义(全流程统一)
- **blocker**：正确性/安全/数据问题,或违反本文件硬性约定 —— 必须修。
- **major**：明显缺陷、缺测试、重要边界未处理 —— 必须修。
- **minor**：风格、命名、可读性等建议 —— 不阻塞,交用户定夺。

### 工作流
1. **方案** · `planner` 产出方案，写入 `.ai/<yyyy-mm-dd>/<task-slug>/plan.md`，**须包含测试用例**。
2. **方案确认闸【人工】** · 向用户呈现 plan.md 摘要，等待用户确认或修改后再继续；未确认不得进入实现。
3. **实现** · `developer` 实现并自测，确保 plan.md 中的测试用例全部通过，将实际改动、测试结果、与方案的偏差写入到 `.ai/<yyyy-mm-dd>/<task-slug>/implementation-round-<NN>.md` 文件中。
4. **评审** · 运行 `.ai-workflow/scripts/review.sh`，由 Codex 评审当前改动（含新增文件），按严重度列出问题，输出到 `.ai/<yyyy-mm-dd>/<task-slug>/review-round-<NN>-<pass|fail>.md` 中。
    > 实现层存在任一 blocker 或 major 即 `fail`；仅有 minor 判 `pass`；方案/架构层 blocker 单独升级，不计入修复轮次。
5. **修复循环** · 依据评审结果分流:
    - 含 **plan 级 blocker**（方案/架构层面的错，非实现 bug）→ **不进入修复循环**，升级人工或退回步骤 1 重做方案。
    - 含实现层 **blocker / major** → `developer` 进入下一轮（**轮次数 + 1**），逐项修复并将记录写入 `implementation-round-<NN>.md`，随后执行步骤 4，生成同轮次的评审文件。**最多 3 轮**；第 3 轮仍未通过则停止，交人工判断。
    - 仅剩 **minor** → 将 minor 列入步骤 6 交用户定夺。
6. **汇报** · 向用户汇报：改动摘要、评审结论与所用轮次、遗留 minor 问题清单；并**询问是否提交，提交须经用户确认**。
