# 0003:评审轮次上限改为可由用户放宽的默认值

- 日期:2026-07-18
- 背景:plan 评审(plan-review.sh)与实现层修复轮此前均为固定 **3 轮**上限
  (前者脚本硬编码,后者散落于 rawf skill 与 CLAUDE.md 的文字约束)。
  固定值在复杂任务上偏紧:用户明知任务需要更多自动收敛轮次,也只能被迫
  转人工。上限的本意是防自动评审空转,而非限制用户意志。
- 决定:两处上限从"固定值"改为"**默认值**"——用户没有特殊要求时仍为
  3 轮;用户**明确要求**放宽时,以用户指定轮数为准。
  - 载体:当前任务 `plan.md` frontmatter 的两个**可选**字段(正整数):
    `plan_review_max_rounds`(plan 评审)与 `impl_fix_max_rounds`
    (实现修复轮),缺省均为 3。
  - 强制面:plan 评审由 `plan-review.sh` 的 `resolve_max_rounds` 解析并
    机器强制(只认 frontmatter 块内行首字段;缺失 → 3;非正整数 → exit 3);
    实现修复轮维持文字约束(rawf-implement / rawf-review 按同一取值规则
    分流,字段缺失或非法一律按默认 3)。
  - 防滥用:字段**只能**在用户明确提出后写入或修改,写入后须在面向用户的
    方案摘要或实现记录中声明;AI 不得以"预计轮次不够"为由自行写入或
    建议性写入。达上限仍不决转人工的兜底不变。
  - 被否掉的备选:环境变量 / 脚本参数传入上限。否因:无痕、易被顺手设置
    绕过约束;frontmatter 随任务目录入库,授权痕迹可审计,且脚本与 skill
    本就读 plan.md,不新增载体。
- 影响:
  - 默认路径零变化:不写字段时行为与旧版完全一致,plan 模板不加新字段,
    避免默认路径噪音。
  - 涉及面:plan-review.sh、三个 rawf skill、CLAUDE.md、AGENTS.md、
    README.md 的固定"3 轮"表述统一改为默认语义;0001 已追加沿革条目。
  - 残留风险:实现修复轮上限仍无机器强制,与工作流其余文字约束同一信任
    层级;若日后需要根治,可在 review.sh 增加轮次闸,届时新增决策取代本条。
  - 旧任务的 plan.md 无新字段,新脚本按缺省 3 处理,无迁移成本。
