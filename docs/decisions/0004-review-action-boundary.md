# 0004:实现层评审动作边界(静态检查 + 受限单元测试)

- 日期:2026-07-18
- 背景:实现层评审(review.sh)以 `--sandbox workspace-write` 在一次性
  副本中运行 Codex,评审指令要求"实际运行 plan.md 中的测试用例"——
  评审者理论上可安装依赖、build、启动服务,既慢又不可控(dev server
  不自终止,会把后台评审吊死到超时)。
- 决定:为评审者划定动作边界,权威定义在
  `.ai-workflow/review-standards.md`「评审动作边界」小节:
  - **按命令性质判定**,不以运行器名称一刀切:允许运行同时满足
    自终止、不启动服务或长驻进程、不访问网络三条性质的检查命令
    (lint、类型检查、限定到单元子集的 vitest run / pytest);
    单条命令默认 5 分钟超时,超时中止并记录;混合套件只跑可明确
    限定的单元子集。
  - **禁止**:依赖安装、构建(build)、启动任何服务器或长驻进程、
    E2E、访问网络。
  - **边界外用例的证据契约**:实现者(rawf-implement 第 5 步)必须在
    评审前把 build / E2E / 需启动服务类用例的完整原始输出落任务目录
    `evidence/`;评审者不重跑这类用例,只审证据并与实现记录交叉比对,
    证据缺失或矛盾按虚报测试处理(blocker)。
  - 约束为 prompt 层软约束;review.sh 不改,一次性副本隔离与
    workspace-write 沙箱默认断网继续作为机器兜底——越界的最坏结果是
    该轮评审变慢或作废,主工作区不受影响。
- 被否掉的备选(A:read-only 纯静态):把 review.sh 沙箱降为
  `--sandbox read-only`,机器强制评审只读(单测/构建因需写缓存而
  天然跑不起来)。否因:评审将完全失去独立重跑测试的能力,"抓虚报
  测试"退化为纯证据审查;lint / tsc 等虽可运行,核验力度损失大于
  收益。用户在两案间明确选择了保留单元测试重跑能力的 B 案。
- 影响:
  - 评审耗时与资源占用上界收紧,后台评审不再可能被 dev server 类
    命令吊死。
  - build / E2E 类用例的核验力度从"评审者重跑"降为"证据审查",由
    实现者证据契约(完整原始输出强制落 evidence/)补足。
  - 实现者负担略增:边界外用例不再限于"体量较大"才落盘。
  - 残留风险:边界是 prompt 层软约束,评审者可能尝试越界命令;由副本
    隔离与沙箱断网兜底,不伤主工作区。若日后需要机器强制,可为评审
    副本引入命令白名单包装或降沙箱等级,届时新增决策取代本条。
