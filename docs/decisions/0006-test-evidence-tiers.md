# 0006:测试证据契约改为 A/B/C 档位制,部分取代 0005 的统一证据协议

- 日期:2026-07-20
- 背景:0005 决定评审只读直审、不重跑测试,测试真实性全靠证据核验,并把
  证据协议定为「全部测试用例统一」走完整原始输出。运行一段时间后暴露出
  结构性误判:协议只承认一种证据形态(命令 + 完整 stdout/stderr + 退出码),
  而 plan 的测试用例表并不声明「这条怎么验、留什么证据」。证据契约缺位,
  评审端只能对全部用例统一套用最严标准——凡靠视觉、剪贴板、系统弹窗等
  人工交互验证的用例,实现者只能交截图或手工汇总,必然被判「证据缺失」
  而 blocker 打回;而截图在只读沙箱中评审者根本无法读取,是最弱的一种
  证据。典型案例:2026-07-20 settings-style-unify 任务,TC-01 至 TC-08
  全部因「未提供可复核的完整原始执行输出」被打回,实际卡点并非实现质量。
  这不是评审者跑偏,而是规则只给了它一把一刀切的尺子。
- 决定:证据形态改为在 **plan 阶段**以契约形式逐条声明,评审按契约核验——
  - plan.md 测试用例表新增「档位」与「证据形态」两列,分 A/B/C 三档:
    A = 可由非交互命令判定者(测试、lint、tsc、grep、脚本),证据为命令 +
    完整原始输出 + 退出码;B = 靠静态事实即可确证者,证据为日志片段或代码/
    文档引用(须带 `文件:行号`);C = 确实无法自动化的人工交互验证,证据为
    实现者陈述 + 关键锚点;
  - **A 档强制、不得降档**;**C 档 ≤ 用例总数 1/3**(总数 < 3 时至多 1 条)
    **且须写明不可自动化的理由**;截图不得作为任一用例的唯一证据;plan
    未声明档位的用例(历史任务)回退按 A 档处理;
  - 评审逐档核验:A 档缺失/不完整 → blocker 并开补交清单(沿用 0005);
    B 档引用不存在 → blocker,存在但不支持结论 → major;**C 档默认采信
    实现者的结果描述,不得仅以「无原始命令输出」为由判 blocker**,只在
    自相矛盾、plan 中并非 C 档(擅自降档)或关键锚点缺失时判问题;
  - 实现侧对称加固,防止档位制被用作偷懒借口:**不得下调 plan 已声明的
    档位、不得改写用例重要性**;无法按原档位执行时结果记 `blocked` 并停下
    问用户,不得自行降档送审;禁止虚构测试结果;存在 blocked 项不进评审;
  - 被否掉的备选:**比例配额制**(「需证据的用例 ≤ 总数 50%」)——配额与
    证据真实性无因果关系,且可通过拆细用例稀释分母,反而给「少留证据」
    发通行证;真正该约束的是「可自动化却偷懒」,故改为按档位强制。
- 影响:
  - 权威定义落 `.ai-workflow/review-standards.md` 新增的「测试证据档位」
    小节;`prompts/review.md` 改为按档位核验,`prompts/plan-review.md`
    新增证据契约检查点(未逐条声明档位 / 可自动化项塞进 C 档 / C 档超限
    或缺理由,均为 major);
  - 模板与 skill 同步:`templates/plan.md` 加档位与证据形态列,
    `templates/implementation-round.md` 加档位列与 `blocked` 取值,
    rawf-plan / rawf-implement / rawf-review 三个 skill 分别承接声明、
    留证与分流职责;
  - **0005 的证据协议条款被本条部分取代**:其「决定」中「全部测试用例
    统一走证据打回协议」与「影响」中「rawf-implement 第 5 步升级为所有
    用例完整原始输出强制落盘」两处,以本条为准。0005 的其余决定——取消
    副本、`--sandbox read-only` 直审、评审者不运行任何测试、完整性指纹
    双保险——**继续完全有效**,不受本条影响;
  - 残留风险:C 档采信条款可能被滥用,把该自动化的项塞进 C 档。缓解为
    A 档强制 + 1/3 上限 + 逐条理由三重约束,均落在 plan 阶段由 plan-review
    拦截,实现阶段的「不得降档」堵住事后规避;若日后出现滥用案例,新增
    决策收紧(如 C 档需用户逐条确认)。
